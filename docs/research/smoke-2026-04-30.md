## Why deferred

Phase J2 of the implementation plan calls for a small real-world fetch +
synthesize + draft run on a chosen topic to record numbers, costs, and
anomalies. Executing it from this implementation session would require:

1. A live `ANTHROPIC_API_KEY` for Haiku (cards) and Sonnet (synth + draft)
   ÔÇö the SDK reads it from the environment via `new Anthropic()` in
   `src/research/llm/client.ts`.
2. Either a `BRAVE_API_KEY` / `TAVILY_API_KEY` for `web-search`, or
   curated arXiv / seed-urls for the topic.
3. Outbound HTTP from this machine to `export.arxiv.org` / `api.search.brave.com`
   / target article hosts, governed by `robots.txt` and the per-domain rate
   limiter.

This implementation session was scoped to producing a complete, hermetic,
test-passing branch ÔÇö running the smoke against live LLMs and the open
internet would mix in keys, billing, network, and content that don't
belong in the source-of-truth commit history.

## Hermetic coverage already in place

- `test/research/e2e/end-to-end.test.ts` ÔÇö runs `fetchTopic` Ôćĺ

Phase D / E5 / F2 / G2 each ship pipeline-level integration tests that
exercise the same code paths the smoke would, just with mocks at the
LLM/HTTP/SDK boundaries.

## When to run the real smoke

When the user (or a maintainer with API keys + a target topic in mind)
wants to validate cost estimates, prompt quality, and rate-limit
behavior against the open web. Suggested first run:

```sh
hwicortex research topic new rag-eval --from-prompt "Evaluating RAG systems"
hwicortex research fetch rag-eval --max-new 5
hwicortex research synthesize rag-eval
hwicortex research draft rag-eval --prompt "Brief survey of RAG evaluation" --style report
hwicortex research status rag-eval --json
```

Record the output of each command (JSON form) and the resulting card /
note / draft markdown files into a follow-up smoke doc dated for that
run. Note any anomalies (rate-limit retries, robots.txt blocks, schema
errors from the LLM, budget halts).

## What is NOT validated by deferring

- Real Brave / Tavily / arXiv response shape stability over time.
- Real Anthropic prompt-cache hit rates.
- Real-world cost-per-topic numbers.
- Real robots.txt and rate-limit interactions.

These should all be checked the first time a maintainer runs the smoke
against real services.
